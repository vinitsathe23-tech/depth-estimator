V-Model Phases Explained

Left Side: Definition
Right Side: Verification
Key Principles

🔽 Descending Phase - Definition & Design

1. Requirements Analysis (REQ)

What: Define what the system should do

Components: SYS-REQ (System), SWE-REQ (Software)

Artifacts: Requirements specifications, use cases

Phase Goal: Create complete and clear requirements

2. System Design (SWE2)

What: Design the high-level architecture

Components: Architecture, component interactions, interfaces

Artifacts: Architecture documents, design specifications

Phase Goal: Create blueprint for implementation

3. Detailed Design (SWE3)

What: Design each component in detail

Components: Data structures, algorithms, interfaces

Artifacts: Detailed design documents, component specs

Phase Goal: Ready for implementation

4. Implementation (SWE4)

What: Write the code

Components: Source code, documentation

Artifacts: Code, inline documentation

Phase Goal: Build working software

🔼 Ascending Phase - Verification & Validation

5. Unit Testing

What: Test individual components

Verifies: Detailed Design (SWE3)

Artifacts: Unit test cases, unit test reports

Phase Goal: Ensure components work correctly

6. Integration Testing (SWE5)

What: Test component interactions

Verifies: System Design (SWE2)

Artifacts: Integration test cases, integration reports

Phase Goal: Ensure components work together

7. System Testing (SWE6)

What: Test complete system

Verifies: Software Requirements (SWE-REQ)

Artifacts: System test cases, system test reports

Phase Goal: Ensure system meets all requirements

8. Acceptance Testing (VAL)

What: Test user acceptance

Validates: System Requirements (SYS-REQ)

Artifacts: Acceptance test cases, validation reports

Phase Goal: Ensure right thing is built

🎯 Key Principles of the V-Model

✓ Traceability

Every requirement is traced through design to code to tests. Changes at any level are reflected throughout.

✓ Verification

Ensures that what is built is built correctly. Design is verified by code, code is verified by unit tests.

✓ Validation

Ensures that the right thing is built. Requirements are validated by acceptance tests.

✓ Quality Gates

Each phase has defined entry and exit criteria. Only proceed when phase is complete and verified.

✓ Bidirectional Verification

Verification flows both directions: down the left side through design and up the right side through testing.
